한 줄 결론

되돌리는 '강도' 차이예요. soft=HEAD만 뒤로(변경은 대기방에), mixed=대기방까지(기본값, 변경은 작업방에), hard=작업방까지 싹 버림(복구 불가 주의).

아직 커밋을 올리기 전 상태라면 soft·mixed로 변경을 살려 다시 고칠 수 있어요. 반면 --hard는 작업방의 수정까지 버리니, 정말 버려도 되을 때만 쓰세요.

한 줄 결론

reset은 과거로 시계를 되감는 것(역사를 고쳐쓰기), revert는 '부정'하는 새 커밋을 추가하는 것(역사를 안전하게 보존).

아무도 모르는 내 커밋이면 reset으로 깔끔하게. 이미 push해서 남들이 보는 내역이면 revert로 되돌려야 남들의 역사가 꼬이지 않아요.

→ T11 자세히 보기
한 줄 결론

git revert <커밋>으로 취소 커밋을 만들고 다시 push하세요. 역사를 고쳐쓰지 않아 남들에게 안전합니다.

이미 공개된 내역을 reset+push --force로 고치면 동료의 역사와 충돌이 나요. 혼자 쓰는 브랜치가 아니라면 revert가 정답입니다.

git revert <커밋> → T11 자세히 보기
한 줄 결론

--force는 원격 역사를 내 것으로 덮어써 동료의 커밋을 통째로 날릴 수 있어요. --force-with-lease는 원격이 바뀌었으면 거부해 사고를 막아줍니다.

force가 필요한 상황(rebase·amend 후)이라도 항상 --force-with-lease를 쓰는 습관을 들이세요.

git push --force-with-lease → T09 자세히 보기
한 줄 결론

노출된 비밀은 무조건 폐기하고 새로 발급하세요. 그 다음 git filter-repo로 역사에서 파일을 통째로 씻어내고 다시 push.

역사에서 지운다고 해서 "없던 일"이 되진 않아요 — 이미 복사된 사본이 있을 수 있으므로 비밀번호/토큰 교체가 가장 중요합니다. 앞으로는 .gitignore에 .env를 꼭 적어두세요.

한 줄 결론

충돌 표시(<<</===/>>>)를 지우고 최종 모습으로 직접 고친 뒤, git add → 커밋(또는 --continue)하면 끝.

같은 줄을 양쪽이 다르게 고쳤을 때만 충돌이 나요. 어느 쪽이 맞는지(또는 둘 다 섞을지)는 사람이 판단합니다. 편집기나 GitHub의 충돌 해결 UI가 도와줘요.

아직 안 올렸다면 git commit --amend로 직전 커밋의 메시지(또는 내용)를 고쳐요.

이미 push한 커밋이라면 amend 후 --force-with-lease push가 필요하고, 남들과 공유 중이면 amend 대신 새 커밋으로 처리하는 게 안전합니다.

git commit --amend → T03 자세히 보기
한 줄 결론

git reflog로 HEAD가 지나간 흔적을 찾아, 그 커밋 해시로 git reset --hard <해시>하면 되돌아올 수 있어요.

reflog는 '숨은 발자취'라서 지워진 것처럼 보여도 잠시(기본 도달 가능 90일) 남아 있습니다. 다만 --hard로 지운 커밋은 시간이 지나면 정리되니 빠를수록 좋아요.

git reflog → T11 reflog 자세히 보기
한 줄 결론

올바른 브랜치로 가서 git cherry-pick <해당 커밋>으로 가져오거나, 아직 안 올렸다면 그 커밋에서 새 브랜치를 만들면 돼요.

원래(잘못된) 브랜치에서는 그 커밋을 revert/reset으로 정리하면 깔끔합니다.

한 줄 결론

1순위는 비밀번호를 즉시 변경/토큰 폐기. 그 다음 git filter-repo로 역사에서 제거하고 강제 push, 마지막으로 GitHub의 secret scanning 경고도 확인.

역사에서 지워도 누군가 이미 복사했을 수 있으니 '없던 일'로 믿지 마세요. 비밀 교체가 유일한 확실한 조치입니다.

한 줄 결론

branch는 한 저장소 안의 평행 우주, fork는 남의 저장소 전체를 내 계정로 통째로 복사(다른 저장소)하는 거예요.

내 저장소에서 실험은 branch, 남의 프로젝트에 기여할 때는 fork 후 PR이 보통의 흐름입니다.

→ T15 fork 자세히 보기
한 줄 결론

fetch는 원격 변경을 '받아만' 오고(작업방 건드림 없음), pull은 받아서 바로 합치기(fetch+merge)까지 해요.

먼저 뭘 받았는지 살펴보고 싶으면 fetch, 바로 내 작업에 반영하려면 pull. pull은 브랜치가 꼬일 수 있으니 확인 후 쓰면 안전합니다.

→ T09 자세히 보기
한 줄 결론

merge는 두 갈래를 합쳐 병합 커밋을 남기고(역사 보존), rebase는 내 커밋들을 한 줄로 다시 나열해 역사를 깔끔하게 정리해요.

"사실이 남은 기록"이 중요하면 merge, "읽기 좋은 한 줄 역사"가 중요하면 rebase. 단 이미 공유한 커밋은 rebase 금지.

→ T08 트레이드오프 자세히 보기
한 줄 결론

clone은 원격 저장소를 내 컴퓨터로 다운로드(복사), fork는 GitHub 서버 위에서 내 계정로 저장소를 복사하는 거예요.

fork는 보통 그 직후 내 포크를 clone해 로컬에서 작업합니다. fork=서버 복사, clone=내 PC 복사.

→ T15 자세히 보기
한 줄 결론

PR(pull request)은 "제 브랜치 변경을 받아주세요" 하고 리뷰·검사·병합을 거치는 제안이에요. 브랜치를 push하고 GitHub에서 'Compare & pull request'를 누르면 만들어요.

터미널이면 gh pr create 한 줄로도 됩니다. 리뷰→승인→squash/rebase/merge 순으로 병합돼요.

gh pr create → T16 PR 자세히 보기

해당 주제의 질문이 없어요. '전체'를 선택해 보세요.

주의

이미 push해서 남들과 공유 중인 커밋은 reset/rebase/amend로 고쳐쓰지 마세요. 동료 역사가 꼬입니다. 공개 내역은 revert로 안전하게 되돌리세요.
