참조 허브 · R02

용어집 — 두려운 단어 사전

"rebase가 한 줄로 뭔지?" 용어마다 일상 비유 한 문장을 먼저, 그 다음 공식 정의를. 알파벳/주제로 거르고 #branch·#pull-request 앵커로 직접 링크.

주제

repository (저장소)

비유 — 사진첩 그 자체. 지금까지 찍은 모든 사진(버전)이 보관되는 상자.

공식 정의: Git이 변경 이력(커밋들)과 메타데이터를 보관하는 데이터 구조. 작업 디렉토리 + .git 디렉토리로 구성된다 (git-scm.com).

→ T01 자세히

commit (커밋)

비유 — 사진첩에 찍어 붙이는 한 장의 사진. 그 순간의 프로젝트 모습을 영구히 보존.

공식 정의: 작업 트리의 스냅샷에 대한 불변(immutable) 기록 단위. SHA-1 해시로 식별되며 부모 커밋을 가리킨다 (git-scm.com/docs/git-commit).

→ T03 자세히

staging area / index

비유 — 대기방. 다음 사진에 넣을 파일만 골라 모아두는 곳.

공식 정의: 다음 커밋에 포함될 변경을 임시 보관하는 '인덱스' 파일. git add로 채운다 (git-scm.com, 3 영역 중 하나).

→ T04 자세히

working directory (작업 디렉토리)

비유 — 작업방. 지금 내가 직접 고치고 있는 실제 파일들.

공식 정의: 저장소의 체크아웃된 파일들이 놓인 디렉토리. 3 영역(working/staging/repository) 중 실제 편집이 일어나는 공간.

→ T04 자세히

HEAD

비유 — '지금 내가 여기 있어' 손가락. 현재 보고 있는 사진/브랜치를 가리키는 포스트잇.

공식 정의: 현재 체크아웃된 커밋(또는 브랜치)을 가리키는 특수 포인터. 대개 현재 브랜치의 최신 커밋 (git-scm.com).

→ T06 자세히

detached HEAD (분리된 HEAD)

비유 — 브랜치 없이 사진 한 장 위에 둥둥 떠 있는 상태. 새 커밋이 길을 잃을 수 있어 주의.

공식 정의: HEAD가 브랜치 참조가 아닌 특정 커밋을 직접 가리키는 상태. 체크아웃 후 새 커밋은 어떤 브랜치에서도 도달 불가능해질 수 있다.

→ T06 자세히

branch (브랜치)

비유 — 평행 우주. 원본을 그대로 둔 채 안전하게 새 실험을 이어가는 갈래.

공식 정의: 특정 커밋을 가리키는 이동 가능한(movable) 포인터. 새 커밋마다 자동으로 전진한다 (git-scm.com/docs/git-branch).

→ T06 자세히

merge (병합)

비유 — 두 평행 우주의 이야기를 하나로 엮어 한 권의 책으로.

공식 정의: 두 브랜치의 변경 이력을 합쳐 새 커밋(병합 커밋)을 만드는 연산. fast-forward와 3-way 병합 방식이 있다 (git-scm.com/docs/git-merge).

→ T07 자세히

fast-forward merge

비유 — 갈래가 난 적 없는 곧은길. 포인터만 앞으로 쭉 밀어주면 끝.

공식 정의: 대상 브랜치가 분기 이후 갈래 없이 직선 연장일 때, 병합 커밋 없이 포인터만 앞으로 옮기는 병합. 새 커밋을 만들지 않는다.

→ T07 자세히

3-way merge

비유 — 두 갈래와 그 출발점 셋을 겹쳐 비교하며 한 권으로 엮기.

공식 정의: 두 브랜치의 끝 커밋과 공통 조상(베이스) 세 점을 비교해 병합 커밋을 만드는 방식. 갈래가 이미 나뉘었을 때 쓰인다.

→ T07 자세히

conflict (충돌)

비유 — 두 사람이 같은 문단을 서로 다르게 고쳐 어느 쪽이 맞는지 책임자가 결정해야 할 때.

공식 정의: 병합/리베이스 시 같은 줄을 양쪽이 다르게 변경해 Git이 자동 합치지 못하는 상태. 수동으로 해결 후 커밋해야 한다.

→ T07 자세히

rebase (리베이스)

비유 — 내 사진들을 뽑아 다른 우주 끝으로 옮겨 다시 차례대로 붙이기. 역사를 한 줄로 깔끔하게.

공식 정의: 한 브랜치의 커밋들을 다른 베이스 위로 옮겨 재적용하는 연산. 역사를 선형으로 정리 (git-scm.com/docs/git-rebase).

→ T08 자세히

interactive rebase (-i)

비유 — 과거 사진들을 편집기에서 한 줄씩 손보기. 합치기·고치기·순서 바꾸기 가능.

공식 정의: git rebase -i로 커밋별 작업(pick/squash/reword/reorder/drop)을 지정해 역사를 재구성하는 모드.

→ T08 자세히

squash (스쿼시)

비유 — 여러 장의 잔사진을 한 장으로 뭉쳐 붙이기. "이 작업 하나였어요"로 정리.

공식 정의: 여러 커밋을 하나로 합치는 것. interactive rebase의 squash 명령이나 PR 병합 시 squash merge로 수행.

→ T08 자세히

cherry-pick

비유 — 다른 나무의 열매 한 알만 콕 골라 따오기.

공식 정의: 다른 브랜치의 지정한 커밋(들)을 현재 브랜치에 새 커밋으로 재적용하는 명령 (git-scm.com/docs/git-cherry-pick).

→ T12 자세히

stash

비유 — 하던 일을 잠시 서랍에 넣어두기. 급한 일 끝나면 다시 꺼내 계속.

공식 정의: 커밋하지 않은 변경을 임시 스택에 보관하는 명령. git stash pop으로 복원 (git-scm.com/docs/git-stash).

→ T12 자세히

bisect

비유 — 반씩 쪼개며 범인을 찾는 이진 탐색. "이 버전은 괜찮아 / 이건 안 돼"를 반복.

공식 정의: 좋은/나쁜 커밋을 표시하면 이진 탐색으로 회귀를 유발한 커밋을 찾아내는 명령 (git-scm.com/docs/git-bisect).

→ T12 자세히

remote (원격)

비유 — 내 컴퓨터 밖, 세상 어딘가에 있는 사진첩 사본(GitHub 등).

공식 정의: 로컬이 아닌 원격 호스트의 저장소. origin 등 이름으로 연결한다 (git-scm.com/docs/git-remote).

→ T09 자세히

origin

비유 — '본점'. 내가 clone해 온 원본 원격 저장소에 붙인 기본 이름.

공식 정의: clone 시 자동으로 생성되는 기본 원격 이름. 다른 이름으로 여러 원격을 둘 수 있다.

→ T09 자세히

fork

비유 — 남의 요리책을 통째로 내 책장으로 복사. 내 책에서 마음대로 연습하고 원작자에게 제안.

공식 정의: GitHub에서 다른 사용자의 저장소를 내 계정로 복사하는 기능. 원본과 연결된 상태로 자유롭게 변경 (docs.github.com).

→ T15 자세히

pull request (PR)

비유 — "제가 만든 변경을 받아주세요" 하고 문 앞에 놓고 두드리는 제안서.

공식 정의: 한 브랜치(또는 포크)의 변경을 대상 브랜치에 병합해달라고 요청하는 GitHub의 협업 단위. 리뷰·검사·병합 흐름 (docs.github.com/pull-requests).

→ T16 자세히

issue

비유 — 마을 게시판에 붙이는 '할 일·버그·질문' 쪽지.

공식 정의: GitHub 저장소에서 작업·버그·토론을 기록하는 추적 단위. 라벨·마일스톤·담당자로 관리 (docs.github.com/issues).

→ T16 자세히

label (라벨)

비유 — 이슈·PR에 붙이는 색깔 분류 스티커(bug, enhancement 등).

공식 정의: 이슈/PR을 분류·우선순위화하기 위한 태그. 색상과 이름을 가진다.

→ T16 자세히

milestone (마일스톤)

비유 — "v1.0 출시까지"처럼 이슈·PR을 한 목표로 묶는 이정표.

공식 정의: 특정 목표/버전에 해당하는 이슈·PR들을 묶는 그룹화 단위. 진행률을 추적한다.

→ T16 자세히

tag (태그)

비유 — 특정 사진에 붙이는 이름표. "이 지점이 v1.0!".

공식 정의: 특정 커밋을 가리키는 고정 참조. lightweight와 annotated(서명·메시지 포함) 두 종류 (git-scm.com/docs/git-tag).

→ T10 자세히

annotated tag

비유 — 이름표에 메시지와 도장까지 찍은 정식 버전 표시.

공식 정의: 메시지·작성자·날짜·(선택)GPG 서명을 담은 Git 객체로 만들어지는 태그. 릴리스에 권장.

→ T10 자세히

semantic versioning (SemVer)

비유 — "주.부.수" 세 숫자로 변화의 크기를 알려주는 버전 관례(예: 2.1.0).

공식 정의: MAJOR.MINOR.PATCH 형식의 버전 번호 체계. 각 자리는 호환성 깨짐/기능 추가/수정을 의미 (semver.org).

→ T10 자세히

submodule

비유 — 책 속에 다른 책 한 권을 통째로 끼워 넣기.

공식 정의: 저장소 안에 다른 Git 저장소를 특정 커밋으로 참조·삽입하는 기능 (git-scm.com/docs/git-submodule).

→ T12 자세히

worktree

비유 — 한 저장소를 여러 책상에 펼쳐놓고 각각 다른 장(브랜치)을 동시에 작업.

공식 정의: 같은 저장소의 여러 작업 디렉토리를 동시에 사용하는 기능. 브랜치 전환 없이 병렬 작업 (git-scm.com/docs/git-worktree).

→ T12 자세히

Dependabot

비유 — 의존성을 지켜보는 로봇 경비. 취약한 패키지를 알리고 고쳐주는 PR을 올려요.

공식 정의: GitHub의 의존성 관리 봇. alerts(취약점 알림)와 security updates(수정 PR 자동 생성) 두 기능 (docs.github.com/code-security).

→ T19 자세히

Actions workflow

비유 — "push하면 자동으로 점검·배포해줘"라며 미리 짜둔 로봇 작업 지시서.

공식 정의: .github/workflows/*.yml에 정의된 자동화 절차. 이벤트·잡·스텝·러너로 구성 (docs.github.com/actions).

→ T18 자세히

runner (러너)

비유 — 작업 지시서를 실제로 수행하는 일꾼 기계.

공식 정의: GitHub Actions 잡(Job)을 실행하는 서버(인스턴스). GitHub 호스팅 또는 self-hosted.

→ T18 자세히

matrix (매트릭스)

비유 — "Node 20·22, OS mac·linux"처럼 조합마다 일꾼을 한 번에 여럿 보내는 격자.

공식 정의: 하나의 잡을 변수 조합(Node/OS 버전 등)별로 병렬 펼쳐 실행하는 Actions 전략 (strategy.matrix).

→ T18 자세히

secret / variable

비유 — secret은 금고 속 비밀번호(로그에도 안 보임), variable은 바꾸기 쉬운 설정값.

공식 정의: secret은 마스킹되어 저장되는 민감값(토큰 등). variable은 평문 설정값. 둘 다 Actions/워크플로우에서 참조 (docs.github.com/actions/security-guides).

→ T18 자세히

branch protection (브랜치 보호)

비유 — 중요 방(main)에 자물쇠. 검사 통과·리뷰 승인 없이는 아무나 직접 넣지 못하게.

공식 정의: 특정 브랜치에 강제 검사(필수 리뷰, 상태 검사, 서명 등)를 설정해 직접 push/병합을 제한하는 규칙 (docs.github.com/repositories/configuring-branches).

→ T17 자세히

CODEOWNERS

비유 — "이 폴더는 이 팀이 책임져요"라고 적어둔 담당자 배정표.

공식 정의: 경로 패턴별 소유자(팀/사용자)를 지정해 PR에 자동 리뷰어로 지정하는 파일 (docs.github.com/repositories/managing-your-repositorys-settings-and-features/customizing-your-repository/about-code-owners).

→ T17 자세히

조건에 맞는 용어가 없어요. 필터나 검색어를 바꿔보세요.
