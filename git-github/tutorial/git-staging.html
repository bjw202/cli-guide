튜토리얼 · T04 · Git 심화

Git에는 방이 3개 있어요

T01에서 "방이 3개 있다"고만 짚고 넘어갔죠. 이번엔 그 방의 정체를 제대로 파헤쳐요. 작업 방, 대기 방, 기록 방 — 이 세 방이 어떻게 연결되고, 왜 '대기 방(staging)'이 필요한지 알아봐요.

이렇게도 써요

디자이너도 작업이 한꺼번에 끝나지 않아요. 오늘은 로고, 내일은 색 팔레트, 모레는 타이포. 변경을 주제별로 나눠 묶어 두면 나중에 '이때 로고만 롤백하자'처럼 정교하게 되돌리기 쉬워요. Git의 대기 방은 바로 그 '주제별 묶음'을 준비하는 자리예요.

세 방의 이름과 역할

Git은 파일을 셋으로 나눠 봐요. 각 방에 이름이 있어요.

  • 작업 방(working directory) — 지금 내가 직접 고치고 있는 곳. 에디터로 파일을 열어 타이핑하는 그 공간이에요.
  • 대기 방(staging area) — '다음 사진에 넣겠다'고 표시한 파일들이 모이는 곳. git add를 치면 이 방으로 파일이 들어와요.
  • 기록 방(repository) — 사진첩 그 자체. git commit으로 도장을 찍으면, 대기 방의 내용이 이 방에 영구히 한 장으로 남아요.

파일은 이렇게 흘러요: 작업 방 → (add) → 대기 방 → (commit) → 기록 방. 한 방향으로만 흐르고, 중간의 '대기 방'이 왜 필요한지는 곧 알게 돼요.

비유

대기 방(staging)은 '사진을 찍기 전에 인물을 정렬하는 무대' 같아요. 무대에 올린 사람만 사진에 찍혀요. 올리기 전엔 얼마든지 다시 빼고, 무대에 없는 사람은 이번 사진에서 빠진 채로 찍히죠.

왜 굳이 '대기 방'이 필요할까?

"작업 방에서 바로 기록 방으로 가면 안 되나요?" 하는 의문, 자연스러워요. 대기 방이 필요한 이유는 한 사진에 무엇을 넣을지 정교하게 고를 수 있기 때문이에요.

예를 들어 오늘 로고도 고치고, 오타도 고치고, 설정 파일도 손봤어요. 이걸 한 사진에 다 때려 넣으면, 두 달 뒤에 "이때 뭘 바꿨더라?" 했을 때 섞여서 알아보기 힘들어요. 대기 방에서 로고만 골라 한 사진으로 찍고, 그다음 오타만 골라 또 한 사진으로 찍으면 — 사진첩이 '로고 바꿈', '오타 수정'처럼 주제별로 깔끔하게 정리돼요.

즉 대기 방은 '주제별 묶음을 준비하는 프레임'이에요. T03에서 배운 git add는 사실 이 방에 파일을 보내는 명령이었고, git commit은 이 방의 내용을 사진으로 찍는 명령이었어요.

팁

한 사진에 너무 많은 변경을 한 번에 넣지 마세요. 주제별로 나눠 찍으면 나중에 되돌리기 쉽고, 동료가 리뷰하기도 편해요. "이 사진은 로고만" "이 사진은 문서만"처럼 한 일을 한 사진에 담는 습관이 숙련자의 비결이에요.

세 방 사이의 '차이'를 보는 도구 — diff

세 방이 있으니, "어느 방과 어느 방을 비교할까?"에 따라 git diff가 여러 모습으로 쓰여요. 아래 터미널은 대기 방과 기록 방(마지막 사진) 사이의 차이를 보는 예시예요.

터미널 — diff 변형 비교
스크롤하면 재생됩니다…

기본 git diff는 작업 방 vs 대기 방을 비교해요. "뭘 더 대기 방에 안 넣었나?"를 보여주죠. 반면 git diff --staged(또는 git diff --cached, 같은 뜻)는 대기 방 vs 마지막 사진을 비교해요. "이번에 넣으려는 게 뭔가?"를 보여줘요.

다른 자주 쓰는 변형들:

  • git diff HEAD — 작업 방 vs 마지막 사진 (전체 차이).
  • git diff --stat — 줄 단위가 아니라 파일별 요약(몇 줄 바뀌었는지)으로.
  • git diff <파일> — 특정 파일만 비교.
  • git diff --word-diff — 줄이 아니라 단어 단위로 바뀐 점을 표시.
  • git diff branch1..branch2 — 두 브랜치(T06) 사이의 차이.
알아두면 좋은

--staged와 --cached는 완전히 같은 뜻이에요. Git의 오래된 이름이 'cache'였던 흔적이 남아 있어서 두 단어 모두 쓸 수 있어요. 요즘은 의미가 더 명확한 --staged를 더 많이 써요.
