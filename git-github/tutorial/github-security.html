안전하게 함께 일하는 법 — GitHub 보안

여럿이 일하면 좋은 점도 많지만, 실수가 미치는 영향도 커져요. GitHub는 한 번의 실수로 모든 게 무너지지 않도록, 여러 겹의 안전망을 제공해요.

이렇게도 써요

공공 기관·정부 부처도 GitHub로 협업할 때, 바로 이런 안전망들에 의존해요. 감사 추적, 권한 통제, 취약점 자동 알림까지 한 번에 갖추는 셈이에요. 혼자 쓰는 사이드 프로젝트에도 그대로 적용된답니다.

여러 겹의 안전망을 상상해 보세요

한 겹의 두꺼운 벽보다, 여러 겹의 얇은 안전망이 더 안전할 때가 있어요. 한 겹이 뚫려도 다음 겹이 막아주니까요. GitHub 보안은 크게 네 겹으로 볼 수 있어요.

  • 계정 겹: 비밀번호, 2FA로 내 계정 자체를 보호.
  • 저장소 겹: branch protection, 권한 설정으로 '누가 뭘 할 수 있는지' 통제.
  • 코드 겹: Dependabot, code scanning이 코드와 의존성의 취약점을 검사.
  • 커밋 겹: commit signing으로 "이 커밋은 정말 내가 한 거예요" 증명.

이 네 겹이 서로 보완해서, 한 가지 실수 하나로는 큰 사고가 안 나도록 해요.

비유

Dependabot은 밤새 도서관을 돌아다니며 '이 책엔 오타가 있어요'라며 알려주는 성실한 경비원이에요. 내가 직접 찾아다니지 않아도, 의존성 패키지에 취약점이 발견되면 알아서 알려줘요.

경비원 Dependabot — 알림과 자동 수정은 달라요

내 코드는 직접 짰지만, 활용하는 패키지(남이 만든 도구)는 수십~수백 개예요. 그중 하나에 취약점이 발견되면? Dependabot이 알아서 알려줘요. 다만, 알림과 자동 수정은 서로 다른 기능이에요. 헷갈리기 쉬운 부분이니 잘 구분해 두세요.

구분 Dependabot alerts Dependabot security updates
하는 일 취약점을 알려주기만 함 고치는 PR을 자동으로 만들어 줌
내 역할 알림을 보고 직접 결정 열린 PR 검토 후 병합만 하면 끝
형태 Security 탭의 경고 목록 자동 생성된 Pull Request
별도 설정 기본 알림만 켜짐 설정에서 따로 켜야 함

"알아서 알려줘"까지만 원할 수도 있고, "고쳐주는 PR까지 만들어 줘"까지 원할 수도 있어요. 저장소 Settings → Security에서 원하는 수준을 고르면 돼요.

비밀을 코드에 적었다면 — secret scanning

토큰이나 API 키를 코드에 그대로 적고 push해버렸다면? GitHub의 secret scanning이 그 모양을 알아보고 즉시 알려줘요. 공개 저장소라면 누구나 그걸 볼 수 있으니 정말 위험해요. 알림을 받는 즉시 비밀번호를 바꾸고 역사에서 지워야 해요.

비슷하지만 다른 기능으로 code scanning이 있어요. secret scanning이 '비밀번호가 새어나갔는지'를 찾는다면, code scanning은 CodeQL 같은 도구로 '코드 자체에 취약한 패턴이 있는지'를 검사해요. SQL 주입, 버퍼 오버플로 같은 흔한 취약점을 찾아줘요.

주의 — 정말 중요해요

한 번 공개 저장소에 push한 비밀번호는 '지웠다'고 안전하지 않아요. Git 불변성 때문에 역사에 남아요 (T01 참고). 즉시 그 비밀번호/토큰을 폐기하고 새로 발급받아야 해요. 그 다음 git filter-repo로 역사에서 제거하는 작업이 필요해요. GitHub에 '비밀 노출' 경고가 뜬다면 절대 무시하지 마세요.

main 브랜치는 함부로 만지지 마세요 (branch protection)

main 브랜치는 보통 '지금 실제로 쓰이는 코드'예요. 여기에 아무나 직접 push하면 실수 하나로 서비스가 멈출 수 있어요. branch protection은 이런 일을 막아줘요. 대표적인 규칙들:

  • 직접 push 금지: 무조건 PR을 거쳐야만 병합 가능.
  • 리뷰 필수 (required reviews): 최소 1명 이상의 승인이 있어야 병합.
  • 상태 검사 필수 (status checks): CI 테스트가 통과해야 병합 (T18과 연결).
  • 브랜치 최신화: 병합 전 main의 최신 변경을 가져와야 허용.

Settings → Branches에서 'Add branch protection rule'로 설정해요. 조직 단위로 정책을 일괄 적용할 수도 있어요.

// branch protection rule — GitHub 설정의 핵심 항목
// Settings → Branches → Add rule
Require a pull request before merging: ON
  Required approving reviews: 1 이상
Require status checks to pass: ON
  CI(test, lint) 통과 필수
Require branches to be up to date: ON
Do not allow bypassing the above settings: ON

이 파일은 이 사람이 검토해요 (CODEOWNERS)

branch protection의 '리뷰 필수'와 짝을 이루는 게 CODEOWNERS예요. "이 폴더의 코드는 이 사람이 꼭 검토해야 해"라는 규칙을 적어두면, 해당 파일이 바뀌는 PR엔 자동으로 그 사람이 리뷰어로 지정돼요. 보안이 중요한 폴더(예: 결제 로직)에 딱이에요.

# .github/CODEOWNERS 예시
# 각 줄은 "파일 경로 패턴   담당자"
*.md                    @docs-team                # 문서는 docs 팀
/src/security/          @security-team            # 보안 코드는 보안 팀
/package.json           @tech-leads               # 의존성 변경은 팀장

계정의 기본 안전장치 — 2FA

2FA (2단계 인증)은 비밀번호 외에 두 번째 확인(전화 앱의 코드, 지문 등)을 요구하는 거예요. 비밀번호가 유출돼도 공격자는 두 번째 단계를 넘지 못해요. GitHub는 모든 계정에 2FA 켜기를 강력히 권장하고, 일부 작업(패키지 게시 등)에는 필수로 요구해요. 보안의 가장 기본이면서 가장 효과적인 단계예요.

"이 커밋, 정말 내가 한 거예요" (commit signing)

누군가 내 이름으로 가짜 커밋을 올린다면? Git 자체는 작성자 이름을 그냥 믿어요. 이걸 막는 게 commit signing이에요. 커밋에 '서명'을 붙여서 진짜 내가 만든 거라 증명하는 거예요. GitHub는 서명된 커밋에 'Verified' 배지를 달아줘요. 서명 방법은 크게 두 가지예요.

구분 SSH 서명 GPG 서명
기술 SSH 키 (이미 push에 쓰던 것과 같은 기술) GPG 키 (전통적인 서명 전용 도구)
설정 난이도 비교적 간단 (SSH 키가 이미 있으면 더 쉬움) 조금 더 복잡 (키 생성·등록 절차)
추천 요즘 GitHub가 권장하는 방식 오래된 방식이지만 여전히 지원

두 방식 모두 결과는 같아요: 커밋이 'Verified' 표시를 받고, 누군가 내 이름을 도용해도 서명까지 위조하긴 매우 어려워요.

Secrets과 Variables — 또 한 번 정리

T18에서 잠깐 다뤘던 두 단어를 보안 관점에서 다시 볼게요. 이 둘의 차이는 '암호화 여부'예요. 이 차이를 모르면 큰 사고로 이어질 수 있어요.

구분 Secrets Variables
저장 형태 암호화 저장 평문(일반 글) 저장
로그 노출 절대 찍히지 않음 (자동 마스킹) 로그에 그대로 보임
언제 쓰나 토큰·비밀번호·API 키 URL·포트 번호·환경 이름
참조 ${{ secrets.이름 }} ${{ vars.이름 }}

"비밀번호인데 Variables에 넣어도 되나?" → 절대 안 돼요. 비밀은 무조건 Secrets에. 이 한 가지만 기억하셔도 사고를 절반은 막을 수 있어요.

알아두면 좋은

GitHub의 Security 탭에 가면 한 저장소의 보안 상태가 한눈에 들어와요. Dependabot 경고, secret scanning 결과, code scanning 결과가 다 모여 있어요. 매일 볼 필요는 없지만, 일주일에 한 번쯤 들여다보면 '빚'이 쌓이지 않아요.
