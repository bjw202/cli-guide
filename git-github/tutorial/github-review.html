코드 리뷰, 서로 키워주는 시간

좋은 리뷰는 칭찬이자 훈련이에요. '틀린 걸 잡는' 자리가 아니라 '서로 더 나은 코드로 키우는' 자리 — GitHub의 리뷰 도구를 차근차근 만나봐요.

이렇게도 써요

교사가 학생 과제에 달아주는 피드백도 코드 리뷰와 같아요. 오픈소스에서도 전 세계 낯선 사람끼리 이 방식으로 리뷰해요. 리뷰는 전문가만의 게임이 아니라, '서로 키워주는' 보편적인 대화 방식이랍니다.

교사 피드백 사례 보기 → 오픈소스 리뷰 사례 보기 →

리뷰는 '잡는' 게 아니라 '키우는' 시간

코드 리뷰(code review)는 다른 사람이 쓴 코드를 읽고 의견을 나누는 시간이에요. 흔히 '틀린 걸 찾아내는' 자리로 생각하지만, 진짜 목적은 달라요. 서로 더 나은 코드로 키우는 자리예요. 질문 하나, 제안 하나가 코드의 미래를 바꿔요.

좋은 리뷰 문화에서는 초보자의 PR도 환영받고, 사소한 실수도 부드럽게 알려줘요. '틀렸다'고 지적하는 게 아니라 '이러면 더 좋을 것 같아요'라고 제안하는 거예. 말투가 달라지면 같은 코드도 다르게 자라요.

비유

리뷰는 동네 북클럽과 같아요. 누군가 쓴 글(코드)을 모여서 읽고, '이 부분 좋더라', '여긴 이렇게 바꾸면 더 읽기 쉬울 것 같아'라고 이야기하는 거예요. 작가를 깎아내리는 게 아니라, 다음 글을 더 잘 쓰도록 돕는 자리랍니다.

리뷰 도구: comments와 suggestions

GitHub의 PR 화면에서 코드는 한 줄씩 보여요. 리뷰어는 review comments(리뷰 코멘트)로 코드 특정 줄에 코멘트를 달아요. "이 변수 이름이 헷갈려요"처럼요. 줄 단위로 달기 때문에 어느 부분에 대한 말인지 명확해요.

더 강력한 도구가 suggestions(제안)이에요. 리뷰어가 "이렇게 바꾸면 어떨까요?" 하고 코드를 직접 제안하면, 작성자는 클릭 한 번으로 그 제안을 적용할 수 있어요. 복사-붙여넣기 없이, 제안 그대로 커밋에 반영돼요. 수정이 부드럽게 이어지는 비결이에요.

CODEOWNERS: 이 구역은 내 담당

저장소가 커지면 "이 폴더는 A팀, 저 폴더는 B팀"처럼 담당을 나누게 돼요. 이걸 자동으로 처리하는 게 CODEOWNERS예요. 저장소 루트의 .github/CODEOWNERS 파일에 '이 경로는 이 사람이 리뷰한다'를 적어두면, 해당 경로가 바뀌는 PR은 자동으로 그 사람을 리뷰어로 지정해요.

예를 들어 이렇게 적을 수 있어요.

# .github/CODEOWNERS 예시
# 문서 폴더는 문서팀이
/docs/           @docs-team
# 데이터베이스 관련 코드는 DBA가
/src/db/         @dba-alice @dba-bob
# package.json은 프론트 리드가
package.json     @frontend-lead

branch protection: 중요 줄기 잠그기

혼자 쓸 땐 아무나 main 브랜치에 push해도 돼요. 하지만 팀이나 오픈소스라면, main 브랜치를 함부로 못 건드리게 잠가야 해요. 이게 branch protection rules(브랜치 보호 규칙)이에요. 설정에서 'main 브랜치 보호'를 켜면 이런 규칙을 정할 수 있어요.

자주 쓰는 보호 규칙 세 가지를 알아둬요.

  • required reviews(필수 리뷰) — 병합하려면 최소 N명의 승인(approve)이 있어야 함. 한 명도 안 보고 병합하는 일을 막아요.
  • status checks(상태 검사) — CI(자동 테스트, T18에서 배워요)를 통과해야만 병합 가능. 빨간 불이면 병합 버튼이 안 눌러요.
  • 직접 push 금지 — main에 곧바로 commit을 올릴 수 없음. 반드시 PR을 거쳐야 함.

이 규칙을 '설정' 같은 곳에 적어 두는 게 아니라, GitHub가 강제로 지키게 해 줘요. 한 번 켜두면 아무도 슬쩍 빠져나갈 수 없죠.

// branch protection rule — GitHub 설정의 핵심 항목
{
  "required_status_checks": {
    "strict": true,              // 최신 main 기준으로 검사
    "contexts": ["ci/tests"]     // 통과해야 할 CI 작업
  },
  "required_pull_request_reviews": {
    "required_approving_review_count": 1  // 최소 1명 승인 필수
  },
  "enforce_admins": true,        // 관리자도 규칙 예외 없음
  "restrict_pushes": true        // main 직접 push 금지
}

리뷰 코멘트는 '사람'이 아니라 '코드'에 남겨요. "이 부분은 X한 이유로 Y하면 좋겠어요"가 "이거 왜 이래?"보다 백 번 낫습니다. 질문 형태("여기 이렇게 한 이유가 궁금해요")는 방어적이지 않게 대화를 여는 가장 부드러운 방법이에요.

리뷰 흐름: comment → resolve → approve → merge

리뷰는 대부분 이런 흐름으로 흘러가요.

  1. comment — 리뷰어가 코드에 코멘트나 제안을 달아요.
  2. resolve — 작성자가 코멘트를 반영하거나 답변하고, 'Resolved' 체크로 해결 표시를 해요.
  3. approve — 리뷰어가 '좋습니다' 승인을 보내요.
  4. merge — 작성자(또는 관리자)가 PR을 병합해요.

모든 코멘트가 resolve되고 승인이 떨어지면, 그제서로 병합 버튼이 눌러요. branch protection이 켜져 있다면 이 흐름을 건너뛸 수 없죠. 그래서 코드가 안전하게 자라는 거예요.

알아두면 좋은

리뷰에도 에티켓이 있어요. 큰 PR보다 작은 PR이 리뷰하기 편하고, 제목과 설명이 명확하면 리뷰어 시간이 아껴져요. 칭찬도 잊지 말기 — "이 부분 깔끔하네요" 한마디가 리뷰 문화를 따뜻하게 만들어요.
