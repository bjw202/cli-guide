내가 자리 비울 때 대신 일하는 로봇 — GitHub Actions

누군가 저장소에 변경을 올리면 알아서 테스트를 돌리고, 통과하면 알아서 사이트를 배포하는 '로봇 비서'가 있다면 어떨까요? GitHub Actions이 바로 그 로봇이에요.

이렇게도 써요

개인 블로그도 글을 push하면 자동으로 사이트가 다시 만들어지고, 팀은 코드를 올릴 때마다 자동으로 테스트를 돌려요. 이 페이지가 보이는 이 사이트도 GitHub Actions로 배포됐을지 모른답니다.

로봇 비서를 상상해 보세요

여러분이 자리를 비운 사이에, 누군가 문을 두드리면 (이벤트) 알아서 일을 처리하는 로봇 비서가 있다고 상상해 보세요. GitHub Actions은 바로 그런 로봇이에요. "이런 일이 일어나면, 이렇게 해 줘"라고 미리 적어두면, GitHub가 그 약속대로 대신 일해요.

IFTTT나 Zapier 같은 자동화 도구를 써보셨다면, 그것의 'GitHub 버전'이라고 생각하면 딱 맞아요. 다만, GitHub Actions은 코드 저장소에 아주 자연스럽게 붙어 있어서, 개발뿐 아니라 문서 사이트 배포나 정기 작업까지 두루 쓰여요.

비유

요리 레시피를 생각해 보세요. "손님이 오면(이벤트), 주방에서(east 환경), 재료 손질 → 굽기 → 접시에 담기(단계들)를 순서대로 해 줘." GitHub Actions의 workflow 파일이 바로 로봇에게 건네는 이 요리 레시피예요.

레시피는 YAML이라는 글로 적어요

로봇 비서에게 건네는 레시피는 .yml 확장자를 가진 YAML 파일로 적어요. 저장소 안의 .github/workflows/ 폴더에 넣으면 GitHub가 알아서 읽어요. 크게 네 부분으로 짜여 있어요.

# .github/workflows/deploy.yml — 로봇 비서에게 주는 레시피
name: 사이트 배포                    # 이 워크플로우의 이름

on:                                  # 1) events — 언제 로봇을 부를까?
  push:
    branches: [main]                 # main에 push할 때

jobs:                                # 2) jobs — 어떤 작업 묶음을 할까?
  build-and-deploy:
    runs-on: ubuntu-latest           # 4) runners — 어디서 돌릴까? (GitHub 제공 가상머신)
    steps:                           # 3) steps — 한 job 안의 단계
      - uses: actions/checkout@v4    # 남이 만든 액션(marketplace) 가져다 쓰기
      - run: npm install             # 직접 명령 실행
      - run: npm run build
      - run: npm run deploy

레시피의 네 가지 재료

위 파일에서 events / jobs / steps / runners 네 단어를 봤어요. 이게 GitHub Actions의 전부랍니다.

  • events (on): "언제 로봇을 부를까?" — push, pull_request처럼 누군가 무언가를 했을 때도 있고, schedule(정해진 시간)이나 workflow_dispatch(내가 직접 버튼 누름)처럼 시간/사람 주도일 수도 있어요.
  • jobs: "어떤 큰 작업 단위로 나눌까?" — 예컨대 '빌드하기' job과 '배포하기' job을 나눌 수 있어요. job끼리는 순서를 정할 수도 있고, 동시에 돌릴 수도 있어요.
  • steps: "한 job 안의 작은 단계들" — 첫째 줄, 둘째 줄… 순서대로 실행돼요. uses로 남이 만든 액션을 가져오거나, run으로 직접 명령을 적을 수 있어요.
  • runners (runs-on): "이 작업을 어디서 돌릴까?" — ubuntu-latest, windows-latest, macos-latest 같은 GitHub 제공 가상머신이 가장 흔해요. 우리 컴퓨터가 아니라 GitHub 클라우드의 컴퓨터에서 돌아간다고 생각하면 돼요.
알아두면 좋은

Marketplace Actions는 남이 만들어둔 레시피 조각을 가져다 쓰는 '마트'예요. actions/checkout@v4(저장소 코드 가져오기), actions/setup-node@v4(Node.js 설치)처럼 흔한 작업은 누군가 이미 만들어뒀어요. uses: 한 줄이면 끝나요. 바퀴를 다시 발명할 필요 없답니다.

여러 버전에서 한 번에 돌리기 (matrix)

내 코드가 Node 20에서도, Node 22에서도 잘 돌아가는지 확인하고 싶다면 어떨까요? 하나하나 따로 돌릴 필요 없어요. matrix를 쓰면 여러 조합을 한 번에 복사해서 동시에 돌려요.

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node-version: [20, 22]      # Node 20과 22에서 각각 한 번씩
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
      - run: npm test

로봇에게만 살짝 알려주는 비밀 (secrets)

배포하려면 서버 비밀번호나 토큰이 필요하죠. 그런데 그걸 코드에 직접 적으면 누구나 볼 수 있어요. GitHub는 secrets라는 '금고'를 제공해요. 저장소 설정에서 한 번 넣어두면, 로봇만 꺼내 쓰고, 로그에는 절대 찍히지 않아요.

반면, 값은 있지만 비밀이 아닌 설정값(예: BASE_URL = https://example.com)은 variables에 넣어요. 둘의 차이는 '암호화 여부'예요.

steps:
  - run: deploy --token ${{ secrets.DEPLOY_TOKEN }}   # 금고에서 꺼내 쓰기
  - run: build --base-url ${{ vars.BASE_URL }}        # 평문 설정
주의

secret은 절대 코드에 직접 적지 마세요. GitHub 설정의 Secrets에 넣고, 워크플로우에서 ${{ secrets.이름 }}으로 참조만 하세요. 한 번 공개 저장소에 적힌 비밀번호는 역사에 남아서 '지웠다'고 안전하지 않아요 (T19에서 자세히).

결과물 보관하고, 다음엔 빠르게 (artifacts, caching)

로봇이 만들어 낸 결과물(예: 빌드된 앱)을 artifacts로 보관하면, 나중에 다운로드하거나 다음 단계로 넘길 수 있어요. 그리고 매번 똑같이 다운로드하는 의존성 패키지는 caching으로 한 번 저장해두면 다음 실행이 훨씬 빨라져요.

예를 들어 npm 패키지를 캐싱하면 두 번째 실행부터는 설치 시간이 몇 분에서 몇 초로 줄어들어요. 흔히 쓰는 캐시는 Marketplace에 이미 만들어진 액션(actions/cache@v4)으로 한 줄이면 끝나요.

dev, staging, prod — 환경을 나눠요 (environments)

보통 실제 서비스는 개발(dev) → 검증(staging) → 실제(prod) 환경으로 나눠요. GitHub Actions의 environments를 쓰면, 각 환경마다 다른 비밀을 두고, prod 환경은 사람이 '승인' 버튼을 누르기 전엔 진행되지 않게 막을 수도 있어요. 실수로 운영 서버에 배포하는 사고를 막는 안전장치예요.

두 글자 CI, 두 글자 CD

GitHub Actions에서 자주 듣게 될 말이 CI/CD예요. 한 번에 풀면:

  • CI (Continuous Integration, 지속적 통합): 코드가 올라올 때마다 자동으로 테스트를 돌려요. "이 변경, 괜찮은 거 맞아?"를 로봇이 확인해 주는 거예요.
  • CD (Continuous Deployment/Delivery, 지속적 배포): 테스트를 통과하면 자동으로 실제 서비스에 배포해요. "사람 손이 필요 없어"가 핵심이에요.

이 두 가지를 합쳐서 'CI/CD 파이프라인'이라 불러요. GitHub Actions은 이 파이프라인을 만드는 도구 중 하나예요.

이 사이트도 이렇게 배포돼요 (GitHub Pages)

지금 읽고 계신 이 안내서 같은 정적 사이트는, main 브랜치에 push하면 자동으로 빌드되어 GitHub Pages로 배포되게 만들 수 있어요. 별도 서버가 필요 없어요. 글을 고치고 push하면 잠시 뒤 사이트에 반영되는 '마법'이 Actions 덕분이에요.

GitHub Pages 설정에서 "Source: GitHub Actions"를 고르고, 간단한 워크플로우 파일 하나면 끝나요. 블로그·이력서·문서 사이트를 무료로 운영할 수 있는 가장 흔한 방법이에요.

터미널에서 GitHub 다루기 (gh CLI)

GitHub를 매번 웹브라우저에서 클릭하기 귀찮을 때가 있어요. gh CLI는 터미널에서 GitHub를 다루는 명령어예요. PR 만들기, 이슈 목록 보기, Actions 실행 확인까지 전부 키보드로 할 수 있어요.

gh pr create                    # 지금 브랜치로 PR 만들기
gh issue list                   # 저장소 이슈 목록 보기
gh run list                     # 최근 Actions 실행 결과 보기
gh workflow run deploy.yml      # 워크플로우 수동 실행(workflow_dispatch)
옛날 이야기

2019년 GitHub Actions이 발표되기 전엔, 자동화를 위해 Travis CI, Jenkins, CircleCI 같은 외부 서비스를 따로 연결했어요. Actions이 나온 뒤로는 '저장소 안에 자동화가 있는' 세상이 됐어요. 설정 파일 하나만 있으면 어디서든 같은 로봇이 일해요.

runners (runs-on)ubuntu/windows/macos 가상머신 — GitHub 클라우드.
