튜토리얼 · T08 · 브랜치·병합

사진첩의 순서를 다시 정리하기 — 리베이스

병합은 합치기만 할 뿐 순서를 안 건드려요. 리베이스(rebase)는 '내 사진들을 다른 우주의 맨 끝으로 다시 옮겨 놓기'예요. 역사가 한 줄로 깔끔해지지만, 조심할 점도 있어요.

이렇게도 써요

큰 팀에서는 'feature 브랜치 → develop → main'으로 단계를 나누는 GitFlow라는 작업 흐름을 써요. 이 흐름이 깔끔하게 보이려면 리베이스로 사진들을 한 줄로 정리하는 게 거의 필수예요. 리베이스는 이런 체계적인 작업 방식의 친구랍니다.

사진들을 다른 우주의 맨 끝으로 옮겨 놓기

T07의 병합은 두 우주를 합쳤지만, 역사 모양은 그대로 둬요. 두 갈래가 합류하는 모양이 그대로 남죠. 가끔은 역사를 '처음부터 한 우주에서 쭉 이어진 것처럼' 깔끔하게 보이게 하고 싶을 때가 있어요. 이게 바로 리베이스(rebase)가 하는 일이에요.

리베이스는 '내 브랜치의 사진들을 떼어서, 다른 브랜치의 맨 끝에 다시 차례로 붙이는' 작업이에요. 마치 내 실험들을 '상대방이 이미 해 놓은 일' 뒤로 모두 옮겨 놓는 거예요. 결과는 한 줄의 깔끔한 역사가 돼요.

비유

병합은 두 갈래 길이 합류점을 새로 만드는 거고, 리베이스는 내 길을 통째로 들어 올려 상대 길의 끝에 이어 붙이는 거예요. 둘 다 결국 같은 지점에 닿지만, 길의 모양이 다르게 남아요.

기본 리베이스

기본 명령은 git rebase <브랜치>예요. 지금 있는 브랜치의 사진들을 떼어서, 그 브랜치의 맨 끝으로 재배치해요. 예를 들어 feature 브랜치에 있다가 git rebase main을 치면, 내 작업이 main의 최신 사진 바로 뒤로 옮겨 붙어요.

주의할 점이 있어요. 사진을 '다시 옮겨 붙이면' 그 사진의 지문(SHA-1)이 바뀌어요. 사진의 내용은 같아도 '다른 사진'이 되는 거예요. 이 점이 다음 절의 핵심 경고로 이어져요.

주의 (가장 중요!)

이미 남에게 보여준(공개한, push한) 브랜치는 리베이스하지 마세요. 다른 사람이 이미 그 사진들을 보고 작업하고 있을 수 있어요. 사진을 '다시 쓰면' 지문이 바뀌어 남들의 사본과 엇갈리고, 큰 혼란이 와요. 리베이스는 '아직 나만 아는' 사진에만 쓰는 규칙이에요.

대화형 리베이스: 사진들을 직접 손질하기

git rebase -i HEAD~3처럼 -i를 붙이면, 최근 3개의 사진을 한 장씩 어떻게 처리할지 고를 수 있는 목록이 열려요. 이걸 대화형(interactive) 리베이스라고 불러요. 다음 다섯 가지 손질이 가능해요.

pick은 그대로 두기, squash는 앞 사진에 합치기, reword는 메시지만 고치기, 순서를 바꾸면 reorder, 목록에서 지우면 drop이에요. 이 도구 하나면 '커밋 정리'라는 귀찮은 일이 우아해져요.

팁

작업을 여러 조각 사진으로 나눠 올리다 보면, 남에게 보여주기 전에 한 번에 합치고 싶을 때가 있어요. 그럴 때 git rebase -i로 여러 사진을 squash 하나로 모으면 깔끔해져요. 단, push하기 전에만요.

리베이스냐, 병합이냐 — 트레이드오프

둘 다 결국 '같은 지점에 닿게' 하지만 길 모양이 달라요. 리베이스는 역사가 한 줄로 깔끔해요. 단, 사진을 '다시 쓰기'에 이미 남과 공유한 사진에는 위험해요. 병합은 역사가 그대로 보존돼요. 단, 병합 커밋이 섞여 지저분해질 수 있어요.

팀마다 규칙이 달라요. '공개 브랜치는 병합, 내 개인 브랜치는 리베이스'를 쓰는 팀이 흔해요. 팀의 합의된 규칙을 따르세요.

옛날 이야기

'rebase vs merge' 논쟁은 2010년대 Git 커뮤니티의 영원한 주제였어요. '역사는 고쳐 쓰는 게 아니라 사실대로 남겨야 한다'는 쪽과 '역사는 읽기 좋게 정리돼야 한다'는 쪽이 오래 다퉜죠. 결론은 정답 없음 — 팀마다 규칙을 정하는 거예요.

받아올 때도 리베이스로: pull --rebase

남이 올린 변경을 내 저장소로 받아올 때, 보통 병합이 일어나요. 그런데 병합 커밋이 자꾸 쌓여 역사가 지저분해진다면, 받아올 때 리베이스로 처리할 수 있어요. git pull --rebase를 쓰면 받아온 뒤 내 사진들을 그 위로 다시 옮겨 놓아요. 역사가 한 줄로 깔끔하게 유지돼요.

매번 --rebase를 붙이기 귀찮다면, 설정으로 '이 브랜치에서 받아올 때는 항상 리베이스'라고 미리 정해둘 수도 있어요. 다음 장(T09 원격)에서 더 자세히 다뤄요.

알아두면 좋은

리베이스를 하다 막혔을 때도 탈출구가 있어요. git rebase --abort로 시작 전으로 되돌아갈 수 있고, git rebase --continue로 충돌을 풀고 진행할 수 있어요. 충돌 해결은 T07에서 배운 것과 같은 방식이랍니다.

🌿 rebase — 가지를 끝으로 옮기기

'merge'와 'rebase' 단계를 번갈아 보면 차이가 명확해져요. rebase는 역사가 일직선으로 정리돼요.

reorder / drop순서 바꾸기 / 사진 버리기.
