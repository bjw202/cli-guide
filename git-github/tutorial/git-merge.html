튜토리얼 · T07 · 브랜치·병합

평행 우주 합치기 — 그리고 부딪혔을 때

두 브랜치를 하나로 합치는 걸 병합(merge)이라 해요. 대개는 자동으로 되지만, 같은 줄을 서로 다르게 고쳤다면 '충돌'이 나요. 당황하지 마세요 — 해결책이 있어요.

이렇게도 써요

RPG 게이머는 보스 전과 후의 세이브 파일을 따로 둬서, 결과가 마음에 안 들면 예전 지점으로 되돌려요. 병합은 그 두 갈래 진행을 한 줄로 모으는 작업과 같아요. 결과를 합치고도 언제든 이전 갈래로 돌아갈 수 있죠.

두 갈래 길이 하나로 합류할 때

T06에서 평행 우주를 만들어 봤어요. 이제 그 우주에서의 실험이 끝났어요. 원본 우주(main)로 가져와 하나로 합치고 싶죠. 이 작업을 병합(merge)이라고 불러요.

병합은 두 우주의 가장 최근 사진을 한 장으로 합쳐 새 사진을 만드는 일이에요. 합치는 방법은 두 가지 상황으로 나뉘어요. 한쪽만 앞으로 갔을 때, 그리고 양쪽 다 변했을 때요.

비유

병합은 두 갈래 길이 다시 만나 하나로 합류하는 지점을 만드는 일이에요. 한 길만 쭉 뻗어 있으면 그냥 그 길을 따라가면 되고(fast-forward), 두 길 다 각자 뻗었다면 합류점을 새로 만들어야 해요(3-way).

한쪽만 앞으로 갔다면: fast-forward

main은 가만히 있고 브랜치만 앞으로 나아갔다면, 합치기가 아주 쉬워요. main의 이름표를 그냥 브랜치의 맨 끝 사진으로 옮기기만 하면 끝이에요. 새 사진을 만들 필요도 없어요. 이걸 fast-forward(빨리 감기) 병합이라고 불러요. 비유하자면 '한 길만 걸었으니 그냥 그 길로 따라가는' 거예요.

양쪽 다 변했다면: 3-way merge

main도 앞으로 가고, 브랜치도 각자 앞으로 나아갔다면 이야기가 달라져요. 두 끝 사진만 봐서는 '무엇이 바뀌었는지' 알기 어려워요. 그래서 Git은 세 장을 꺼내요 — main의 끝, 브랜치의 끝, 그리고 두 갈래가 갈라진 공통 조상 사진이요.

이 세 장을 비교해 각 변경을 합친 새 사진을 만들어요. 이걸 3-way merge라고 불러요. 비유하면 '두 갈래 길이 만나 합류점을 새로 하나 만드는' 거예요. 이때 만들어지는 사진을 병합 커밋(merge commit)이라 해요.

같은 줄을 서로 다르게 고쳤을 때: 충돌

대개는 Git이 알아서 잘 합쳐요. 그런데 두 우주가 같은 문단의 같은 줄을 서로 다르게 고쳤다면, Git은 어느 쪽이 '맞는지' 스스로 결정하지 못해요. 이때 충돌(conflict)이 나요. 실패가 아니라 '이 부분은 네가 정해줘'라는 부탁이라고 생각하세요.

충돌이 나면 파일 안에 표시가 세 줄씩 찍혀요. 충돌 마커라고 부르는 이 표시들은 이렇게 생겼어요.

function 인사() {
<<<<<<< HEAD
  return "안녕하세요";
=======
  return "반갑습니다";
>>>>>>> feature/greeting
}

표시의 의미는 단순해요. <<<<<<<부터 =======까지가 지금 내 우주(HEAD)의 버전이고, =======부터 >>>>>>>까지가 가져오려는 브랜치의 버전이에요.

비유

충돌은 두 사람이 같은 문단을 각자 고쳤을 때 일어나요. Git은 어느 쪽이 '맞는지' 모르기 때문에 우리에게 결정을 맡기는 거예요. 실패가 아니라 '너 결정해줘'라는 부탁이에요.

충돌 푸는 네 단계

충돌을 만나면 이 순서대로 하면 돼요.

첫째, git merge <브랜치>를 치고 충돌 메시지를 확인해요. Git이 어떤 파일이 부딪혔는지 알려줘요. 둘째, 충돌 난 파일을 열어 마커(<<<<<<<, =======, >>>>>>>)를 찾아요. 셋째, 두 버전 중 하나만 남기든, 둘을 섞든 직접 수정하고 마커를 지워요. 넷째, git add <파일>로 고쳤다고 알린 뒤 git commit으로 병합을 마쳐요.

주의

충돌을 풀다가 길을 잃었다면, git merge --abort를 치면 병합을 시작하기 전으로 안전하게 되돌아가요. 일단 고친 뒤 커밋하기 전이라면 언제든 쓸 수 있어요. 떨지 말고 이 탈출구를 기억하세요.

팁

충돌이 자주 난다면 브랜치를 더 짧게 유지하고 자주 병합하세요. 오래 떨어져 있을수록 같은 줄을 건드릴 확률이 커져요. 작은 브랜치, 잦은 병합이 충돌을 줄이는 가장 효과적인 습관이랍니다.

알아두면 좋은

3-way merge를 할 때 fast-forward처럼 병합 커밋 없이 합치고 싶다면 git merge --ff-only로 fast-forward만 허용하거나, 반대로 항상 병합 커밋을 만들려면 git merge --no-ff를 써요. 팀마다 '흔적을 남긴다/안 남긴다' 취향이 달라요.

⚔️ 충돌 직접 풀어보기

같은 줄을 두 사람이 다르게 고쳤어요. A/B 중 하나를 고르면 결과가 바로 합쳐져요.
