튜토리얼 · T14 · GitHub 협업

Git은 도구, GitHub는 마을

비슷한 이름이라 늘 헷갈리죠. Git은 내 컴퓨터에서 사진첩을 정리하는 '연필'이고, GitHub는 그 사진첩을 모아두고 함께 보는 '인터넷 마을'이에요.

이렇게도 써요

리눅스 커널부터 정부 공공 데이터까지, 수많은 프로젝트가 GitHub라는 마을에 모여 살아요. 혼자 쓰는 도구에서 '함께 만드는 광장'으로 — 버전 관리가 마을이 되면 힘이 커진답니다.

헷갈리는 두 단어부터 정리할게요

사람들은 "GitHub에 올렸어요"라고 말할 때, 사실은 "Git으로 사진첩을 만들고, 그걸 GitHub라는 마을에 올렸어요"라고 말하는 거예요. 두 단어는 붙어 다니지만, 역할이 달라요.

Git은 T01에서 배운 그 사진첩 프로그램이에요. 내 컴퓨터에서 혼자 쓰는 도구예요. 인터넷 없이도 작동해요. 반면 GitHub는 그 사진첩을 인터넷에 올려놓고 다른 사람과 함께 보는 마을이에요. Git 없이 GitHub는 성립하지 않지만, Git만 있다고 마을이 되진 않아요.

비유

Git이 '연필'이라면, GitHub는 '모두가 글을 올리는 게시판'이에요. 연필로 쓴 글을 게시판에 붙이면 남들이 읽고, 복사(fork)하고, 의견을 달고, 함께 고칠 수 있죠. 연필은 혼자 쓰는 도구지만, 게시판은 함께 쓰는 공간이랍니다.

한 표로 비교해 볼게요

두 단어의 차이를 표로 보면 훨씬 명확해요.

GitHub는 '호스팅 플랫폼'이에요

전문 용어로 GitHub는 코드 호스팅 플랫폼(code hosting platform)이에요. '남의 사진첩을 인터넷에 올려주는 창고'라는 뜻이에요. 비유하자면, Git이 '책을 만드는 인쇄소'라면 GitHub는 '그 책을 진열하고 파는 서점'이에요.

이런 창고는 GitHub만 있는 건 아니에요. GitLab, Bitbucket도 같은 역할을 해요. 다만 GitHub가 가장 많은 사람이 쓰고, '소셜' 기능이 풍부해서 대세가 됐어요. 어떤 창고를 쓰든 아래쪽은 Git이라는 도구가 똑같이 깔려 있어요.

마을이 특별한 이유: 소셜 코딩

GitHub는 단순한 창고가 아니에요. social coding(소셜 코딩)이라는 문화를 만든 곳이에요. 코드에 '좋아요'를 누르고(star), 남의 프로젝트를 내 계정으로 복사하고(fork), 고친 걸 제안하고(pull request), 문제를 논의하고(issue) — 코드를 '함께 만드는' 공간이 된 거예요.

이런 대화 기능이 GitHub를 단순 저장소 이상의 협업 광장으로 만들어요. T15에서는 저장소를, T16에서는 Issue와 PR을, T17에서는 코드 리뷰를 차례로 배워요. 마을의 규칙을 하나씩 익히는 셈이에요.

옛날 이야기

GitHub는 2008년 샌프란시스코에서 시작됐어요. 'Git을 더 쉽게 함께 쓰자'는 아이디어였죠. 그 전에는 SourceForge, Google Code 같은 마을이 먼저 있었어요. 2018년에는 Microsoft가 GitHub를 인수했고, 지금은 전 세계 1억 명이 넘는 개발자가 이 마을에 살고 있어요.

한 가지 더: GitHub ≠ Git

가장 흔한 오해 하나. "GitHub에서 배우면 Git을 배우는 거냐"고 물으면 — 반만 맞아요. Git은 도구 그 자체이고, GitHub는 그 도구를 얹은 마을이에요. 마을의 규칙(Pull Request, Issue, branch protection)은 GitHub만의 것이지, Git 자체의 규칙은 아니에요.

그래서 GitLab이나 Bitbucket으로 옮겨도, Git 명령어는 똑같이 쓰지만 마을의 모양은 달라요. 우리는 이 시리즈에서 Git(T01~T13)을 먼저 익혔고, 이제 GitHub라는 마을(T14~T21)의 규칙을 배울 차례예요.

알아두면 좋은

GitHub 말고도 비슷한 마을이 있어요. GitLab(자체 서버 운영에 강함), Bitbucket(Atlassian 제품군과 연동), Gitea(가볍게 직접 설치). 회사에 따라 쓰는 마을이 달라요. 하지만 'Git으로 사진첩을 만든다'는 본질은 모두 같답니다.
