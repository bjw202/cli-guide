대화로 코드를 만드는 법: Issue와 PR

GitHub는 단순한 창고가 아니라 대화 공간이에요. '이거 문제 있어요'라고 말하는 Issue와 '제가 고쳤어요'라고 제안하는 Pull Request, 두 가지 대화 방식을 배워요.

이렇게도 써요

교사도 학생 과제를 Pull Request로 받아요. 오픈소스에 기여할 때도 같은 흐름이 쓰여요. Issue로 문제를 알리고, PR로 해결을 제안하는 — GitHub가 만든 '대화하며 만드는' 방식이죠.

대화하는 두 가지 방법

마을(GitHub)에서 사람들은 두 가지 카드로 대화해요. 하나는 Issue(이슈), '이거 문제 있어요', '이런 기능 있으면 좋겠어요'라고 말하는 카드예요. 다른 하나는 Pull Request(PR, 풀 리퀘스트), '제가 코드를 고쳤는데 같이 볼래요?'라고 제안하는 카드예요.

Issue는 '무엇이 문제인가'를 말하고, PR은 '어떻게 고쳤는가'를 보여줘요. 보통 Issue로 문제를 먼저 알리고, 그걸 해결하는 PR을 나중에 올리는 흐름이에요. 질문→대답의 관계랄까요.

비유

Issue는 마을 게시판에 붙이는 '건의 메모'예요. "이 길에 웅덩이가 있어요"라고 적어 올리는 거죠. PR은 그 웅덩이를 직접 고친 뒤 "제가 고쳤으니 검토해 주세요" 하고 내어놓는 제안서예요. PR이 더 능동적이고, 코드까지 딸려 있죠.

Issue: 문제를 말하는 카드

Issue는 "이거 버그 있어요", "이런 기능 있으면 좋겠어요" 같은 문제 제기 카드예요. 제목, 설명, 댓글이 달리고, 하나의 이슈는 하나의 대화 주제로 묶여요. 누구나 달아도 되고, 해결되면 'close'로 닫아요.

Issue를 체계적으로 관리하는 도구 세 가지를 알아둬요.

  • Issue templates — 버그 리포트, 기능 요청 등 '양식'을 미리 만들어두는 기능. 새 이슈를 만들 때 양식이 자동으로 채워져서, 빠뜨리는 정보가 줄어들어요.
  • labels — bug, feature, help wanted 같은 색깔 라벨. 이슈를 종류별로 한눈에 분류해요.
  • milestones — "v1.0까지 이 이슈들 끝내자"처럼 여러 이슈를 묶어 목표로 삼는 기능. 마감일과 진척도를 한 번에 봐요.

Pull Request: 변경을 제안하는 카드

PR은 '내가 고친 코드를 같이 볼래요?'라는 제안 카드예요. T15에서 배운 fork(또는 브랜치)에서 코드를 고치고 push하면, 그 변경을 원본 저장소로 '제안'하는 게 PR이에요. PR 안에서는 코드가 한 줄씩 보이고, 리뷰어가 코멘트를 달 수 있어요.

PR의 흐름은 대체로 이래요: fork → branch → commit → push → PR → review → merge. 오픈소스 기여의 표준 동선이기도 해요.

PR의 상태는 크게 세 가지로 바뀌어요. 리뷰어가 보낼 수 있는 응답이에요.

  • comment — "이렇게 바꾸면 좋겠어요" 의견만.
  • approve — "좋습니다, 병합해도 돼요" 승인.
  • request changes — "이 부분 고쳐 주셔야 병합 가능해요" 수정 요청.

또 draft PR이라는 것도 있어요. "아직 작성 중인데 미리 보여드려요"라며 피드백만 먼저 구하는 반쯤 완성된 PR이에요. 병합 버튼이 비활성화돼 있어서 실수로 들어갈 일은 없어요.

팁

PR을 열 때 '이 PR이 무엇을 바꾸는지'를 한두 줄로 적어 두면 리뷰어가 훨씬 편해요. 코드만 보여주는 PR보다, '왜 고쳤는지'를 설명하는 PR이 훨씬 빨리 병합된답니다.

충돌이 일어나면: resolve conflicts

PR을 올렸는데, 그사이 다른 사람이 같은 부분을 고쳤다면 충돌(conflict)이 생겨요. T07에서 배운 병합 충돌과 같은 현상이에요. GitHub는 웹 화면에서 간단한 충돌은 바로 고칠 수 있게 해 줘요(resolve conflicts 버튼).

복잡한 충돌은 웹에서 해결이 어려워서, 내 컴퓨터로 가져와서 직접 해결하고 다시 push해야 해요. 충돌 해결은 '두 사람이 같은 줄을 동시에 고쳤을 때 어느 쪽을 택할지 정하는' 작업이에요. T07을 다시 보면 도움이 돼요.

병합(merge)하는 세 가지 방법

리뷰가 끝나고 승인되면, PR을 merge(병합)해요. 이때 GitHub는 세 가지 방식을 제안해요. 설정에서 허용한 방식만 골라 쓸 수 있어요.

방식 무슨 일이 일어나나 언제 어울리나
squash merge 브랜치의 모든 커밋을 하나로 합쳐서 병합. 중간 과정이 사라지고 한 줄로 깔끔하게. 작은 기능, '고치다 만 흔적'을 남기고 싶지 않을 때. 역사를 깔끔하게.
rebase merge 커밋들을 그대로 이어 붙이되, main 줄기 위에 차례로 올림. 병합 커밋이 안 생김. 커밋 하나하나의 의미를 보존하고 싶을 때. 선형 역사 선호.
merge commit 병합 전용 커밋을 하나 만들어서 합치기. 기본값. 두 줄기가 만났다는 흔적이 남음. 팀 표준, '여기서 합쳤다'는 지점을 기록하고 싶을 때.

팀마다 취향이 달라요. "우리 팀은 squash로 깔끔하게", "우리는 merge commit으로 합친 지점을 남겨"처럼 정해서 써요. 중요한 건 한 가지로 통일하면 역사가 일관된다는 거예요.

알아두면 좋은

저장소에 Discussions(질답 게시판), Wiki(문서 모음), Projects(v2)(칸반 보드) 같은 기능도 있어요. Issue는 '문제', Discussions는 '질문', Projects는 '일정 관리'에 어울려요. 같이 쓰면 팀 워크가 체계화된답니다.
