팀마다 다른 Git 사용법, 다섯 가지 철학

Git은 그냥 '도구'예요. 도구를 '어떻게 쓸지'는 팀마다 다 달라요. 똑같은 칼을 두고도, 정교한 식당 주방과 빠른 길거리 노점은 쓰는 법이 다르듯이요. 유명한 다섯 가지 워크플로우를 비교해 볼게요.

이렇게도 써요

Google·Meta 같은 거대 기업은 'Trunk-based'로 수천 명이 한 줄에서 일하고, 명확한 출시 주기가 있는 제품팀은 'GitFlow'로 버전을 관리하고, 한 저장소에 여러 프로젝트를 묶는 모노레포는 또 다른 규칙을 써요. 팀 크기·제품 종류에 따라 어울리는 철학이 달라요.

먼저, 주방으로 비유할게요

같은 '요리'라도, 어디서 하느냐에 따라 방식이 달라요. 정교한 식당 주방은 역할 분담이 세밀하고(셰프·소스 셰프·디저트 담당…), 작은 카페는 한두 명이 빠르게 오가며, 길거리 노점은 한 사람이 모든 걸 빠르게 처리해요. 어느 쪽이 '맞다'가 아니라 상황에 맞는 거예요.

Git 워크플로우도 같아요. 누가 어떤 브랜치에서 일하고, 언제 합치는지에 대한 규칙이 곧 그 팀의 '요리 방식'이에요. 다섯 가지 유명한 방식을 소개할게요.

비유

요약하면: GitFlow는 정교한 식당 주방(역할 분담), GitHub Flow는 스몰카페(빠르고 단순), Trunk-based는 바쁜 길거리 노점(빠른 회전). Forking은 각자 제 주방에서 요리해 와서 진열대에 올리는 재래시장, Release Flow는 식당과 카페를 합친 퓨전 주방이에요.

다섯 가지 워크플로우, 한 표로

말이 길면 헷갈려요. 먼저 한 표로 비교하고, 그 다음 하나씩 풀어 설명할게요.

워크플로우 핵심 언제 어울리나 출처
GitFlow main / develop / feature / release / hotfix 다계층 릴리즈 주기가 명확한 제품 (예: 설치형 앱) Vincent Driessen, 2010
GitHub Flow main + feature 브랜치, 단순 웹 서비스, 잦은 배포 GitHub
Trunk-based 모두가 main에 직접, 짧은 브랜치 대규모, 빠른 통합 (Google·Meta) 대형팀 관행
Forking fork → PR, 원본 저장소 보호 오픈소스 일반적 관행
Release Flow GitFlow + trunk 혼합 (MS) 제품 + 서비스 혼합 Microsoft

GitFlow — 정교한 식당 주방

GitFlow는 브랜치가 여러 층으로 나뉘어요. main(출시용), develop(개발용), feature/*(각 기능), release/*(출시 준비), hotfix/*(긴급 수정). 역할마다 전용 브랜치가 있어서, "지금 이 줄은 무슨 용도인가?"가 명확해요.

장점은 출시 주기가 명확한 제품(예: "1년에 두 번 큰 버전 발표"하는 설치형 소프트웨어)에 잘 맞는다는 거예요. 단점은 배우기 어렵고, 잦은 배포가 필요한 웹 서비스에는 너무 무거워요.

GitHub Flow — 스몰카페

GitHub Flow는 단순해요. main 하나와, 거기서 갈라나온 feature 브랜치만 있어요. 작업이 끝나면 PR로 main에 병합하고 바로 배포. GitFlow의 복잡한 계층은 다 빼고, 'main은 항상 배포 가능해야 한다'는 규칙 하나만 지켜요.

이 방식이 요즘 웹 분야에서 사실상 표준이에요. 작은 팀, 자주 배포하는 웹 서비스, 이 안내서 같은 문서 사이트에 두루 어울려요. 복잡도가 낮아서 처음 도입하기도 좋아요.

Trunk-based — 빠른 길거리 노점

Trunk-based는 'main(trunk)에 모두가 직접 붓는다'는 철학이에요. 브랜치를 아예 안 쓰거나, 쓰더라도 하루 이내로 아주 짧게 써요. Google·Meta 같은 거대 조직이 이 방식으로 수천 명의 작업을 한 줄로 통합해요. 속도가 생명이에요.

단, 이 방식은 '아주 강력한 자동 테스트'가 필수예요. main이 항상 안전한지 CI가 매 순간 검사하지 않으면, 한 사람의 실수가 모두에게 영향을 줘요. 대규모 조직이 아니면 굳이 안 써도 돼요.

Forking — 재래시장

Forking은 원본 저장소를 'fork'(복사)해서 내 계정으로 가져온 뒤, 거기서 작업하고 원본에 PR을 보내는 방식이에요. 원본 저장소의 쓰기 권한을 함부로 주지 않아도 돼서, 오픈소스에서 가장 흔해요. 모르는 사람의 기여도 이렇게 받아요. T15·T16에서 잠깐 봤던 패턴이에요.

Release Flow — 퓨전 주방

Release Flow는 Microsoft가 정리한 방식으로, GitFlow와 Trunk-based의 좋은 점을 섞었어요. main에서 작은 단위로 자주 통합하는 건 Trunk-based처럼, 하지만 출시 시점마다 별도 release/* 브랜치를 만들어 안정화하는 건 GitFlow처럼. '제품(출시 주기 있음) + 서비스(수시 배포)'가 섞인 환경에 어울려요.

옛날 이야기

Vincent Driessen가 2010년 블로그에 올린 "A successful Git branching model"이 GitFlow의 원조예요. 10년 넘게 널리 쓰였지만, 2020년 본인이 "웹 서비스엔 안 맞을 수 있다"고 글을 업데이트했어요. 도구의 창시자조차 상황이 바뀌면 다시 생각한다는 좋은 본보기예요.

알아두면 좋은

요즘 웹 분야에서는 GitHub Flow가 사실상 표준이에요. GitFlow는 출시 주기가 명확한 제품에, Trunk-based는 대규모 조직에 어울려요. 혼자 사이드 프로젝트를 한다면 GitHub Flow로 충분해요. 처음엔 가장 단순한 것부터 시작하고, 팀이 커지면 그때 고민해도 늦지 않아요.

그럼 우리는 뭘 골라야 할까요?

네, 답은 '상황에 따라 다르다'예요. 하지만 출발점은 명확해요. 작은 팀·웹 서비스·잦은 배포라면 GitHub Flow로 시작하세요. 출시 주기가 분명한 제품이라면 GitFlow 또는 Release Flow. 수백 명이 한 저장소에서 일한다면 Trunk-based. 오픈소스라면 Forking이 자연스러워요.

어떤 것을 고르든 핵심은 '우리 팀이 이 규칙을 모두 이해하고 있는가'예요. 화려한 워크플로우를 도입해도 절반만 따르면 혼란만 커져요. 작게 시작해서 모두가 같은 언어로 소통하는 게 훨씬 중요해요.
