# 로드맵 — 향후 구현 Phase 계획

> **PLANNING ONLY — 본 단계에서는 실행하지 않는다.**
> 이 문서는 이후 별도 Phase에서 진행될 구현 계획을 묘사한다.
> 코드는 어디에도 작성되지 않는다. 시간 추정치는 없다 — 우선순위/순서로 표현.

---

## Phase 총괄

```
P1 페이지 골격 & IA 연결
   └→ P2 콘텐츠 + 일러스트
        └→ P3 인터랙티브 데모 & 애니메이션
             └→ P4 QA / 폴리시 / 접근성
```

각 Phase는 이전 Phase의 결과물에 의존한다. 건너뛰기 불가.

---

## P1 — 페이지 골격 & IA 연결 (Priority: High)

**목표**: 단일 HTML 파일의 뼈대를 세우고, content-plan.md의 14개 섹션 구조를 마크업에 반영.

**주요 산출물(방향)**:
- 단일 `index.html` 골격(헤더/푸터/14개 섹션 컨테이너).
- `StickyNav`, `ProgressBar` 컴포넌트 자리잡기(스타일은 최소).
- 14개 섹션의 빈 컨테이너 + 섹션 ID + 내비게이션 연결.
- CSS 변수로 컬러/타이포 토큰 정의(design-guide.md 색상/폰트 매핑).
- 반응형 브레이크포인트 데스크톱/모바일 골격.

**의존성**: 없음(첫 Phase).

**수용 기준(방향)**:
- 브라우저에서 `index.html` 열었을 때 14개 섹션이 순서대로 스크롤됨.
- 내비게이션 클릭 시 해당 섹션으로 부드러운 스크롤.
- 진행률 바가 스크롤에 따라 채움.
- 모바일 뷰포트에서 1단 레이아웃 깨짐 없음.

---

## P2 — 콘텐츠 + 일러스트 (Priority: High)

**목표**: content-plan.md의 카피 방향을 실제 한국어 카피로 다듬어 채우고, 따뜻한 일러스트를 제작/삽입.

**주요 산출물(방향)**:
- 14개 섹션별 본문 한국어 카피 라이팅(비유 선행 원칙 준수 검증 포함).
- side-note 박스 5종(비유/역사/팁/주의/알아두면 좋은) 각 위치에 삽입.
- 비교표/치트시트 표 데이터 채우기(tmux 공식 위키 기반 단축키 검증 재확인).
- 일러스트 세트(방/책상/창문/포스트잇/detach/attach/페어프로그래밍) 제작 또는 라이선스 확보.
- 모든 단축키 `KeyCap` 컴포넌트 적용.
- 이미지 `alt` 작성.

**의존성**: P1 완료.

**수용 기준(방향)**:
- 40개 주제 커버리지 체크리스트 전부 본문에 등장(누락 0).
- 모든 기술 용어 첫 등장 앞에 비유가 선행(오디언스 계약 준수).
- 일러스트가 따뜻한 톤 가이드에 부합.
- 한국어 맞춤법/어투(존댓말 통일) 점검 통과.

## P3 — 인터랙티브 데모 & 애니메이션 (Priority: Medium)

**목표**: design-guide.md의 "핫 에셋" 중 인터랙티브/애니메이션 자산 구현.

**주요 산출물(방향)**:
- `ScrollReveal` — 모든 섹션 리빌 모션(`IntersectionObserver`, `prefers-reduced-motion` 대응).
- `FakeTerminal` 데모 — 버튼 → prefix 키 → pane 분할/detach/attach 시퀀스 애니메이션(순수 JS/CSS).
- `HierarchyDiagram` — server/session/window/pane 건물 비유 스크롤/클릭 확장 모션.
- `CheatSheetCard` — 스티키/토글 단축키 카드.
- `BeforeAfter` — 일반 터미널 vs tmux 슬라이더/탭 비교.
- `DetachReassurance` — FAQ detach 공포 해소 인터랙션.
- Gentle onboarding micro-interactions(호버/키캡 눌림).

**의존성**: P2 완료(콘텐츠 자리 잡힌 상태에서 인터랙션 부착).

**수용 기준(방향)**:
- 모든 데모가 실제 shell 없이 동작(정적 호스팅에서 OK).
- `prefers-reduced-motion` 켜졌을 때 모션 축소/정지.
- 키보드로 모든 인터랙티브 요소 조작 가능(tab/enter/space).
- 모바일 터치 동작 정상.
- 데모가 "거짓말"을 하지 않음 — tmux 실제 동작과 일치(단축키/순서).

## P4 — QA / 폴리시 / 접근성 (Priority: Medium)

**목표**: 전체 마감. 접근성, 성능, 크로스 브라우저, 디테일 폴리시.

**주요 산출물(방향)**:
- WCAG AA 색 대비/스크린 리더/키보드 탐색 전수 점검.
- 크로스 브라우저(Chrome/Safari/Firefox + 모바일 Safari/Chrome) 검증.
- 빌드 단계 없음 확인 — 파일 오픈만으로 동작.
- 성능 점검(이미지 최적화, 모션 부드러움, JS 최소).
- 한국어 카피 최종 교정(맞춤법/어투 일관성).
- 오타/단축키 정확성 최종 검증(공식 위키 대조).
- footer 참고 링크 검증.

**의존성**: P3 완료.

**수용 기준(방향)**:
- Lighthouse 접근성/성능 점수 목표치(방향: 접근성 95+, 성능 90+) 통과.
- 40개 주제 커버리지 재검 — 여전히 누락 0.
- 비개발자 대상 간이 usability 체크(지인 1~2명) — "tmux가 뭔지 설명할 수 있나요?" 질문에 긍정 응답.

## 위험 & 완화 (Risks)

| 위험 | 완화 |
|---|---|
| 인터랙티브 데모가 tmux 실제 동작과 어긋남 | P2/P3에서 공식 위키 기반 단축키/순서 재검증 의무화 |
| 일러스트 톤 가이드 이탈 | design-guide.md 팔레트/스타일 기준을 P2 시작 시 아티스트에게 전달 |
| 단일 페이지가 너무 길어져 이탈 | 진행률 바 + 스티키 내비로 "위치 감각" 유지; 섹션 간 여백으로 호흡 |
| 모션 과잉 → 산만함 | design-guide.md "gentle" 원칙 준수; P4에서 모션 양 조정 |
| 비개발자 대상 어투 붕괴 | P2 수용 기준에 "비유 선행" 자동 검증 항목 포함 |

## 명시적 비-목표 (Non-Goals for 이 Roadmap)

- 다크 모드 우선 지원 ❌ (밝은 튜토리얼 톤 유지; 다크 모드는 향후 선택)
- 다국어(영문 등) ❌ (이번엔 한국어 단일)
- 실제 터미널 통합(WebContainer 등) ❌ (순수 정적, 가짜 데모로 충분)
- CMS/빌드 시스템 ❌ (단일 HTML, 빌드 단계 없음)
- 백엔드/폼/분석 ❌
